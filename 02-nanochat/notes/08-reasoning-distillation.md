# 推理能力训练 & 模型蒸馏

---

## Part 1：推理能力是如何训练的

---

### 什么是推理能力？

```
普通对话模型：
  问："一个班 30 人，女生占 40%，有多少男生？"
  答："18 人。"（直接输出，不展示过程）

推理模型（o1 / R1 级别）：
  问：同上
  答："让我一步步来：
       1. 女生 = 30 × 40% = 12 人
       2. 男生 = 30 - 12 = 18 人
       答案是 18 人。"
```

**关键发现：推理能力是预训练就有的潜力，训练只是激发它。**

```
大模型预训练时见过：
  - 大量数学解题步骤
  - 代码注释和调试过程
  - 论文的推导步骤

→ 模型"潜在地"学会了逐步推理
  但默认不会主动展示这个过程
  训练目标 = 让模型把潜在推理能力"用出来"
```

---

### 方法一：Chain-of-Thought SFT（思维链微调）

```
普通 SFT 数据：
  输入：30 人，女生 40%，多少男生？
  输出：18 人。

CoT SFT 数据：
  输入：同上
  输出：
    女生 = 30 × 40% = 12 人
    男生 = 30 - 12 = 18 人
    答案：18 人。
```

**本质：** 用带推理步骤的数据做 SFT，改变 QKV 注意力模式，让模型在输出答案前先生成推理步骤。

**局限：** 上限被人类数据质量限制，人能写多复杂，模型就能学多复杂。

---

### 方法二：GRPO（DeepSeek-R1 的核心）

不需要 Reward Model，让模型自己探索推理策略。

```python
# GRPO 伪代码
for each prompt:
    # 生成 G=8 个不同回答
    responses = [model.generate(prompt) for _ in range(G)]

    # 规则验证打分（不是神经网络！）
    rewards = [verify(response) for response in responses]
    # 数学题：答案正确 → +1，错误 → -1
    # 代码题：测试通过 → +1，报错 → -1

    # 组内归一化，计算相对优势
    advantages = (rewards - mean(rewards)) / std(rewards)

    # 比平均好的答案 → 增大概率
    # 比平均差的答案 → 减小概率
    loss = -mean(advantages * log_probs)
    loss.backward()
```

**为什么能激发推理？**
```
奖励信号 = "答案对不对"（客观规则验证）
         不是"听起来好不好"（主观 RM）

模型发现：先写推理步骤的回答
         比直接写答案更容易答对
→ 自发学会了"先思考，再回答"！

这就是 DeepSeek-R1 的 "Aha Moment"：
  没有人告诉模型要反思和回溯
  模型在 RL 中自发涌现出这些行为！
```

---

### 方法三：过程奖励（PRM）

不只奖励最终答案，还奖励每一步推理是否正确：

```
结果奖励（ORM）：
  推理：1+1=2, 2+2=5, 5-1=4   ← 中间步骤出错
  答案：4 碰巧答对 → 给高分！（错误推理被强化）

过程奖励（PRM）：
  Step 1: 1+1=2  ✅ → +0.9
  Step 2: 2+2=5  ❌ → -0.5  立即惩罚！
  → 模型必须每步都正确
```

---

### 方法四：Long CoT + RL（o1 / R1 完整流程）

```
Phase 1: 冷启动
  少量高质量 CoT 数据做 SFT
  让模型了解推理链的格式

Phase 2: GRPO / PPO 强化学习
  用数学/代码题做 RL
  奖励 = 答案是否正确
  模型自发探索更好的推理策略

Phase 3: 拒绝采样 SFT
  用 RL 模型生成大量推理链
  只保留最终答案正确的
  用这些数据再做一轮 SFT

Phase 4: 再次 RL
  继续打磨
```

---

### 为什么数学和代码是最佳训练数据？

#### 原因一：答案客观可验证

```
写文章好不好？  → 主观，需要人评判
数学答案对不对？ → 客观，可自动验证 ✅
代码能否跑通？  → 客观，可自动验证 ✅

→ 奖励信号可靠，无需人工或 Reward Model
```

#### 原因二：推理链与答案强绑定

```
普通问题：
  推理链A → 答案"推荐《速度与激情》"  ← 可能对
  推理链B → 答案"推荐《阿甘正传》"    ← 也可能对
  推理链质量和答案正确性没有强绑定

数学问题：
  错误推理 → 必然导致错误答案
  → "优化正确率"= "优化推理质量" ✅
```

#### 原因三：难度可精确分级

```
Level 1: 1+1=?          （太简单，不需推理）
Level 2: 解方程          （入门，学习推理入口）
Level 3: 竞赛数学        （困难，挑战推理边界）
Level 4: 前沿数学猜想    （太难，模型无法学习）

→ 支持课程学习，始终在最佳学习区间（正确率 20%~80%）
```

#### 原因四：推理步骤天然显式

```
数学解题天然有步骤：
  已知 → 设 → 第一步 → 第二步 → 因此

代码天然有逻辑结构：
  # 每行代码都是一步推理

→ 每步推理都是实际生成的 token
→ RL 可以直接优化推理过程
```

#### 原因五：推理能力泛化性强

```
只用数学训练 → 推理能力迁移到所有领域！

DeepSeek-R1 实验：
  仅用数学+代码做 RL
  MMLU 通用知识：85% → 89%   ← 从没见过这类题
  写作质量：7.2 → 7.8 / 10   ← 写作也变好了

原因：推理是"元能力"
  数学训练的"分解问题→逐步推导→验证结论"
  同样适用于医学、法律、策略规划等所有领域
```

---

### 推理为什么需要更长输出？

```
普通模型：
  x → Transformer × 12层 → 答案
  有效计算深度 = 12 层

推理模型生成 1000 个 token 的推理链：
  每生成一个 token 都经过完整 12 层
  → 有效计算深度 = 12 × 1000 = 12000 层！

本质：用更多推理时间换取更高准确率
这就是"测试时计算（Test-Time Compute）"

类比：
  普通模型 = 直觉反应（System 1）
  推理模型 = 深思熟虑（System 2）
```

## Part 2：模型蒸馏

---

### 什么是蒸馏？

```
不用蒸馏（直接训练小模型）：
  7B 模型做 RL 探索推理策略
  → 参数少，探索空间有限，很难自发发现推理策略
  → 效果差

用蒸馏：
  671B Teacher 生成大量高质量推理链
  → 7B Student 直接模仿 Teacher 的输出
  → 不需要自己"发现"推理，直接"照着学"
  → 效果好得多

类比：
  不蒸馏 = 让小学生自己研究微积分
  蒸馏   = 让大学教授写好解题步骤，小学生照着学
```

---

### 形式一：数据蒸馏（最常用）

```
步骤：
1. Teacher（671B）对每道题生成推理链 + 答案
2. 收集 (问题, Teacher输出) 对，构成蒸馏数据集
3. Student（7B）用这些数据做普通 SFT

本质：就是 SFT，只是数据来源是大模型而非人类
```

---

### 形式二：逻辑蒸馏（更精确）

```
普通 SFT 只学"最终输出的 token"：
  Teacher 输出"巴黎" → Student 学：下一个词是"巴黎"

逻辑蒸馏学 Teacher 的完整概率分布（logits）：
  Teacher 分布：
    "巴黎" → 0.85
    "里昂" → 0.08
    "马赛" → 0.05

  Student 学到："巴黎最可能，但里昂也有 8% 的可能"
  → 学到了 Teacher 的知识边界和不确定性
```

```python
# 普通 SFT Loss
loss_sft = cross_entropy(student_logits, true_token_ids)

# 逻辑蒸馏 Loss（KL 散度）
loss_kd = KL_divergence(
    softmax(student_logits / T),   # T > 1 软化分布
    softmax(teacher_logits / T)    # 暗知识更明显
)

# 混合 Loss（实践中最常用）
loss = α * loss_sft + (1-α) * loss_kd
```

**温度 T 的作用（"暗知识"）：**
```
T=1（正常）：  "巴黎"→0.999，"里昂"→0.001  → 信息量少
T=5（软化）：  "巴黎"→0.60，"里昂"→0.25    → Teacher 的不确定性被保留
```

---

### 蒸馏改变了哪些权重？

蒸馏本质是 SFT，**所有权重理论上都会更新**，但各层影响不同：

| 权重位置 | 影响程度 | 原因 |
|---------|---------|------|
| 深层 Attention QKV | ████████ 最大 | 学习 Teacher 的推理注意力模式 |
| 深层 MLP | ██████ 较大 | Teacher 的推理模式需要压缩进来 |
| 浅层 Attention/MLP | ████ 中等 | 基础语言能力已有，微调即可 |
| lm_head | ███ 较小 | 输出格式向 Teacher 对齐 |
| Embedding | █ 最小 | 词汇表共享，基本不变 |

**逻辑蒸馏 vs 数据蒸馏的差异：**
```
数据蒸馏：深层改动多，浅层少，主要影响 Attention
逻辑蒸馏：全层更均匀，对 MLP 影响更大（软标签传递更多知识）
```

---

### 蒸馏的局限性

#### 局限一：容量上限——小瓶装不下大瓶的水

```
Teacher: 671B 参数
Student:   7B 参数（参数比 1:96）

→ 7B 装不下 671B 的所有知识，必然有"压缩损失"

实验数据（MATH 数据集）：
  R1-671B（Teacher）:   97.3%
  R1-7B（仅蒸馏）:      83.0%   差 14 个点
  R1-7B（自己做 RL）:   72.6%   差更多（但蒸馏有效）
```

#### 局限二：泛化性弱——只会"照猫画虎"

```
RL 训练的模型：
  通过大量试错，"理解"了推理本质
  遇到新题型，能自己摸索
  → 泛化能力强

蒸馏的模型：
  见过类似题 → 模仿 Teacher 推理
  遇到新题型 → 不知所措
  → 泛化能力弱

实验证据：
  分布内测试（同类型题）：  蒸馏 ≈ RL
  分布外测试（新题型）：    蒸馏 << RL

蒸馏学到的是"模式匹配"，RL 学到的是"推理能力本身"
```

#### 局限三：受 Teacher 上限约束

```
Teacher 会犯的错，Student 也会犯（继承错误）
Teacher 没有的能力，Student 也没有

对比 RL 训练：
  奖励来自客观验证，不是 Teacher
  理论上可以超越所有人类水平（AlphaGo 的方式）
```

#### 局限四：推理链可能是"幻觉"

```
Teacher 有时输出：
  推理链看起来完整合理
  但某步是"编的"，最终答案碰巧对了

Student 学到：
  "写出看起来合理的推理"
  但内在逻辑不总是真正可靠

验证困难：
  自动验证只能验证最终答案
  无法验证每步推理是否真实正确
```

---

### 蒸馏 vs RL 全面对比

| 维度 | 蒸馏 | RL 训练 |
|------|------|---------|
| 训练成本 | 低（只需 SFT） | 极高（大量采样+更新） |
| 效果上限 | < Teacher | 可以超越 Teacher |
| 泛化能力 | 弱（模式匹配） | 强（真正推理） |
| 分布外表现 | 差 | 好 |
| 数据依赖 | 高度依赖 Teacher 质量 | 只依赖验证器 |
| 推理可靠性 | 可能是幻觉 | 与答案强绑定 |
| 适用场景 | 快速部署小模型 | 追求最强能力 |

---

### 最佳实践：蒸馏 + RL 结合

```
DeepSeek-R1 的方案：

Step 1: 大模型做 RL（R1-671B 获得强大推理能力）
              ↓
Step 2: 用 R1-671B 生成大量推理数据
        （质量远超人工标注）
              ↓
Step 3: 小模型（7B/14B/32B）用数据做蒸馏
        （快速获得接近大模型的推理能力）
              ↓
Step 4: 对小模型再做小规模 RL（可选）
        （在蒸馏基础上进一步提升泛化）

结果：
  R1-7B（仅蒸馏）:    83.0% MATH
  R1-7B（蒸馏+RL）:   87.2% MATH  ← 更好
  R1-671B（纯 RL）:   97.3% MATH

蒸馏是"捷径"，RL 是"正道"，结合使用效果最佳
```

---

## 一句话总结

> **推理能力通过 CoT SFT（告知格式）+ GRPO/PPO 强化学习（自发发现策略）来激发；模型蒸馏让小模型通过模仿大模型推理链抄近路获得能力，主要改变深层 QKV 和 MLP 权重，但受容量上限、泛化性弱、Teacher 上限、推理幻觉四大局限约束；最佳实践是大模型 RL + 小模型蒸馏 + 可选小规模 RL。**
